# AI 竞赛学习助手 - 设计思路文档

## 一、项目背景与目标

### 1.1 问题背景
大学生参加各类竞赛时面临以下痛点：
- 竞赛信息分散在各个官网，查找困难
- 竞赛通知文档格式多样（PDF、网页、Word），信息提取耗时
- 参赛者和指导教师的报名入口不同，容易混淆
- 竞赛信息更新频繁，难以及时跟踪

### 1.2 设计目标
打造一个智能化的竞赛信息管理平台：
- **一站式管理**：集中管理所有竞赛信息
- **智能提取**：AI 自动从文档中提取关键信息
- **角色区分**：根据用户身份提供对应入口
- **操作便捷**：简化信息录入和查询流程

## 二、核心设计理念

### 2.1 用户体验优先
- 界面简洁直观，减少学习成本
- 操作流程精简，减少点击次数
- 即时反馈，让用户了解系统状态

### 2.2 智能化辅助
- AI 提取替代人工录入
- 智能重名检测避免重复数据
- 置信度评估帮助用户判断结果可靠性

### 2.3 灵活可扩展
- 多 AI 提供商支持，避免单点依赖
- 模块化设计，便于功能扩展
- 标准化接口，方便集成

## 三、功能模块设计

### 3.1 用户认证模块

**设计思路**：
- 采用 JWT 无状态认证，简化服务端管理
- 区分三种角色：参赛者、指导教师、管理员
- 角色决定可访问的功能和跳转入口

**关键决策**：
- 选择 bcrypt 加密密码，安全性高
- Token 有效期 30 分钟，平衡安全与体验
- 前端持久化存储，支持页面刷新后保持登录

### 3.2 竞赛管理模块

**设计思路**：
- 竞赛信息结构化存储，便于检索和展示
- 支持多入口 URL（官网、参赛者入口、教师入口）
- 记录来源信息，便于追溯和更新

**核心功能 - 角色跳转**：
```
用户点击"参赛入口" 
  → 获取用户角色 
  → 查询竞赛的对应 URL 
  → 返回跳转链接
```

这是本系统的核心价值：一键跳转到报名入口。


### 3.3 AI 智能提取模块

**设计思路**：
- 支持多种输入源：URL、文件、文本
- 多 AI 提供商架构，提高可用性
- 结果规范化处理，保证数据质量

**提取流程设计**：
```
输入内容 → 内容获取 → AI 分析 → JSON 解析 → 数据规范化 → 置信度评估 → 返回结果
```

**为什么采用多提供商架构？**
- 不同提供商能力不同（文件支持、网页阅读）
- 避免单一提供商故障影响服务
- 用户可根据需求选择最合适的提供商

**置信度设计**：
- 量化 AI 提取结果的可靠程度
- 低于 60% 提示用户人工核对
- 帮助用户快速判断是否需要修正

### 3.4 重名检测模块

**设计思路**：
避免同一竞赛被重复录入，同时支持信息合并更新。

**当前检测规则**：
1. 完全相同
2. 去除标点后相同
3. 包含关系
4. 去除年份/届数后相同
5. 字符相似度 > 80%
6. 去除前后缀后相同

**处理方式设计**：
- **智能合并**：AI 分析新旧数据，取长补短
- **覆盖更新**：直接用新数据替换
- **另建新条目**：确认不是重复后新建

### 3.5 文档管理模块

**设计思路**：
- 支持常见文档格式（PDF、Word、HTML、TXT）
- 文件安全存储，防止路径遍历攻击
- 解析结果缓存，避免重复解析

**文件处理流程**：
```
上传文件 → 格式验证 → 安全存储 → 内容解析 → 返回文本
```

## 四、技术选型考量

### 4.1 后端框架：FastAPI

**选择理由**：
- 异步支持，适合 IO 密集型的 AI 调用
- 自动生成 API 文档，便于调试
- Pydantic 数据验证，减少错误
- 性能优秀，接近 Node.js

### 4.2 数据库：SQLite

**选择理由**：
- 零配置，开箱即用
- 单文件存储，便于备份迁移
- 对于中小规模数据足够高效
- 配合内存缓存提升热点数据访问速度

**为什么不用 MySQL/PostgreSQL？**
- 本项目为单机部署，无需分布式
- 减少运维复杂度
- 数据量可控，SQLite 完全胜任

### 4.3 前端框架：Vue 3

**选择理由**：
- Composition API 代码组织更清晰
- 响应式系统简单直观
- 生态丰富，Element Plus 组件库成熟
- 学习曲线平缓

### 4.4 AI 提供商选择

| 提供商 | 优势 | 劣势 |
|--------|------|------|
| Google Gemini | 功能全面，支持文件和网页 | 需要科学上网 |
| Cloudflare | 快速稳定，免费额度 | 仅支持文本，不需要科学上网 |
| Cerebras | 高性能推理 | 仅支持文本，需要科学上网 |

若没有科学上网环境，则选择Cloudflare


## 五、界面设计思路

### 5.1 整体风格
- 简洁现代，以蓝色为主色调
- 卡片式布局，信息层次分明
- 适当留白，避免视觉拥挤

### 5.2 AI 提取页面

**三步式流程**：
1. 选择 AI 提供商（卡片选择）
2. 选择内容来源（Tab 切换）
3. 输入内容并提取

**结果展示**：
- 关键信息高亮展示
- 一键保存到竞赛库

### 5.3 保存竞赛对话框

**多步骤设计**：
```
表单编辑 → 重名检测 → [合并预览] → 保存成功
```

**重名列表优化**：
- 从复杂卡片简化为简洁列表
- 只显示必要信息：名称 + 匹配类型
- 点击图标跳转详情页查看完整信息
- 减少信息过载，提升决策效率

### 5.4 竞赛列表页面

**功能设计**：
- 搜索框快速定位
- 状态筛选（即将开始/进行中/已结束）
- 分页加载，避免一次加载过多
- 卡片展示关键信息

## 六、安全设计考量

### 6.1 认证安全
- 密码 bcrypt 哈希存储
- JWT Token 短有效期
- 敏感操作需要认证

### 6.2 输入安全
- Pydantic 模型严格校验
- URL 格式验证，防止注入
- 文件类型白名单限制

### 6.3 文件安全
- 文件名重命名，防止路径遍历
- 大小限制，防止资源耗尽
- 哈希校验，检测重复文件

### 6.4 API 安全
- 速率限制，防止滥用
- CORS 限制，只允许指定域名
- 错误信息脱敏，不暴露内部细节

## 七、性能优化策略

### 7.1 缓存策略
- AI 提取结果缓存，相同输入直接返回
- 竞赛列表前端缓存 5 分钟
- 请求去重，避免重复调用

### 7.2 异步处理
- 数据库操作全异步
- AI 调用异步等待
- 文件解析异步执行

### 7.3 前端优化
- 路由懒加载，减少首屏体积
- 请求合并，减少网络开销
- 状态持久化，减少重复请求
